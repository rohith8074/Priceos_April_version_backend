PriceOS Client Architecture Note

End-to-End Agent Working Guide

This document explains how the PriceOS agent system works across onboarding, market setup, pricing, execution, monitoring, and settings changes. It is written for client-facing review, but it reflects the actual runtime structure found in the current Original_priceos codebase.

Live runtime services Setup and market seeding agents Prompt-defined specialist roles
PriceOS has two views of “agents.” The client-facing view is the business role layer, such as CRO, Event Intelligence, Benchmarking, and Channel Sync. Internally, those roles are implemented by a mix of AI specialist agents, deterministic TypeScript services, internal API routes, and MongoDB collections that act as a shared system memory.

1. System Model

How the platform is organized internally

The architecture is easiest to understand as a shared data spine in the middle, with sync and research on the left, and pricing, chat, and execution on the right.

Inbound data and setup

  • Data Sync Agent pulls listings, calendars, and reservations from Hostaway.
  • Onboarding route stores market choice, strategy, and organization settings.
  • Market Research, Marketing, Benchmark, Guardrails generate initial market context.
  • Event Feed Syncer normalizes public event feeds into one internal event format.

Shared Data Spine

Every agent reads from or writes to the same operating memory so pricing decisions remain explainable and repeatable.

Organization MarketTemplate Listing InventoryMaster Reservation MarketEvent BenchmarkData PricingRule EngineRun ChatMessage

Decisioning and action

  • CRO Router turns user questions into structured analysis and proposals.
  • Deterministic pricing pipeline calculates proposals across the calendar.
  • Channel Sync Agent writes approved changes back to the PMS and verifies them.
  • Status and monitoring layer tracks engine freshness, critical findings, and sync outcomes.

3. End-to-End Runtime

What happens step by step

Each step below shows when the agent is invoked, what it receives, which APIs it touches, and what it returns into the platform.

1
Portfolio Connection

PMS credentials are captured and the account is connected

The onboarding flow starts by storing Hostaway credentials and organization progress. Once a live PMS connection exists, the import layer is able to pull real portfolio data instead of demo fixtures.

Invoked when

User starts onboarding and provides Hostaway account details.

Components

/api/onboarding, DataSyncAgent, organization record update.

Input

hostawayApiKey, hostawayAccountId, user/org identity.

APIs and output

Uses Hostaway. Returns connected listings and saves organization credentials.

2
Portfolio Sync

Listings, calendars, and reservations are imported into the shared data spine

The Data Sync Agent performs the first operational data pull so the rest of the system can reason over real inventory, availability, and booking history.

Invoked when

During initial import and during later manual or scheduled sync triggers.

Components

DataSyncAgent.initialImport(), listing sync, reservation sync, inventory sync.

Input

Organization ID, Hostaway credentials, imported listing IDs.

APIs and output

Calls Hostaway /v1/listings, /v1/reservations, and listing calendar endpoints. Writes Listing, Reservation, InventoryMaster.

3
Market & Strategy

The market, currency, timezone, and pricing posture are locked in

When onboarding reaches completion, PriceOS saves the selected market, chooses the guardrail profile for the chosen strategy, and resolves template defaults such as currency and timezone.

Invoked when

User completes the onboarding steps connect → select → market → strategy → complete.

Components

/api/onboarding, Organization, MarketTemplate.

Input

marketCode, selected listings, strategy, settings overrides.

APIs and output

No external API is required here. The result is updated market identity, currency, timezone, and guardrails on the org.

4
Market Seeding

Market intelligence, events, and benchmark context are created

At onboarding completion, PriceOS seeds market intelligence. In demo mode it uses static templates. In live mode it invokes AI setup agents to create live event and benchmark records for the chosen city.

Invoked when

On onboarding completion, or later via /api/market-setup.

Components

Market Research Agent, Marketing Agent, Benchmark Agent, Guardrails Agent, event seeding logic.

Input

Market template city, custom keywords, seasonal notes, property details, date range, current base price.

APIs and output

Uses Lyzr inference and market search prompts. Writes MarketEvent, BenchmarkData, and optional listing floor/ceiling updates.

5
Event Sync

Public event feeds are normalized into one event layer

The Event Intelligence service can pull events from external public feeds and cache them in a unified format so pricing logic does not depend on any single source shape.

Invoked when

During explicit event sync, market refresh, or via EventIntelligenceAgent.fetchAndCacheEvents().

Components

EventIntelligenceAgent, syncEventFeeds(), /api/v1/system/events/sync.

Input

Organization ID, resolved market city, forward window such as 90 days.

APIs and output

Normalizes Eventbrite, Dubai tourism/calendar feeds, RSS feeds, and Ticketmaster support in the syncer. Returns inserted and updated event counts.

6
Context Build

The system assembles a single trusted context package for AI reasoning

Before AI analysis, PriceOS builds a structured context object from portfolio, benchmark, reservation, inventory, and market data. That context is either embedded directly in the chat call or attached as Lyzr global context.

Invoked when

User opens chat, requests analysis, or a context refresh is triggered.

Components

buildAgentContext(), lyzr-context-sync, chat routes.

Input

Property, metrics, reservations, inventory, benchmark, events, revenue, booking pace, selected date range.

APIs and output

Uses MongoDB reads and optional Lyzr context APIs. Produces one JSON truth package for all downstream AI reasoning.

7
CRO Reasoning

Aria interprets the question and routes specialist analysis

The CRO Router agent is the conversational orchestrator. It receives the full system context, frames the request, and can conceptually delegate to specialist roles such as Property Analyst, Booking Intelligence, Market Research, and Price Guard.

Invoked when

User asks for insight, pricing advice, booking analysis, or market context.

Components

CRO Router prompt, specialist prompt contracts, chat APIs.

Input

market_context, analysis window, property, metrics, inventory, reservations, benchmarks, market events, news, demand outlook.

APIs and output

Uses Lyzr inference. Returns JSON with routing, proposals, and client-ready chat response.

8
Pricing Run

The deterministic pricing engine creates operational proposals

For production calendar pricing, the codebase uses a deterministic pipeline over future dates. It reads the shared data spine and creates proposals with guardrail-aware pricing actions.

Invoked when

/api/engine/run-all is called for all active listings or a chosen subset.

Components

runPipeline(), pricing waterfall, rule application, engine run logging.

Input

Listing, PricingRule, InventoryMaster, event signals, strategy settings, floors and ceilings.

APIs and output

No external API is needed. Outputs proposal fields back into InventoryMaster and run records into EngineRun.

9
Approval & Writeback

Approved proposals are pushed to the PMS and verified

Once a proposal is approved, the Channel Sync Agent executes the change against the PMS, then reads back the result to confirm that the update actually landed.

Invoked when

User approves a proposal via /api/proposals/[id]/approve.

Components

Proposal approval route, ChannelSyncAgent.executeProposal().

Input

Proposal ID, listing ID, approved day-level changes, PMS credentials.

APIs and output

Uses Hostaway calendar interval update plus readback verification. Returns success, updated days, verification state, and execution timestamp.

10
Monitoring & Guest Ops

Status, anomaly detection, and guest communication run as support layers

After pricing decisions are made, the platform surfaces operational health, can classify anomalies, and supports guest communication workflows using reservation-aware context.

Invoked when

Agent status screen loads, guest inbox actions occur, or monitoring workflows are triggered.

Components

/api/agents/status, Anomaly Detector prompt, Guest Reply Agent, Conversation Summary Agent.

Input

Engine run freshness, proposal changes, booking snapshots, reservation threads, listing context.

APIs and output

Uses Lyzr inference plus DB context. Returns health state, anomaly scores, reply drafts, summaries, and escalation flags.

5. External Data and APIs

Where each class of information comes from

PriceOS does not rely on one API. It combines PMS truth, AI inference, and public market signals into one operating memory.

Hostaway API

  • Used for listing import, reservation sync, calendar read, and calendar writeback.
  • Main lane for operational truth about the managed portfolio.
  • Also used for verification after price changes are executed.

Lyzr Inference API

  • Used by CRO Router, market setup agents, benchmarking, guardrails, and guest communication agents.
  • Receives structured prompts plus PriceOS context and returns JSON-shaped specialist outputs.
  • Also supports global context attachment for richer agent memory.

Public Event Feeds

  • Normalized by the Event Feed Syncer into the MarketEvent collection.
  • Current syncer includes Eventbrite, Dubai tourism/calendar sources, RSS-based market feeds, and Ticketmaster support.
  • Used to explain peaks, uplift opportunities, and demand spikes.

Market Search Signals

  • Used mainly during live market setup and benchmark generation.
  • Includes search-oriented prompts for local events, news, holidays, comp sets, and market sentiment.
  • Persisted into events and benchmark records so later runs do not need to rediscover everything from scratch.

6. Settings Change Behavior

What happens when market configuration is changed in Settings

This is the most important lifecycle question for a client because it changes which city, currency, timezone, keywords, and demand signals the system should follow.

Immediate effects when Settings are saved
  • /api/organizations updates the Organization record.
  • If marketCode changes and no currency override is supplied, the route pulls the new market’s default currency and timezone from MarketTemplate.
  • The organization immediately starts reporting the new market identity back to the frontend.
  • Any future context build will see the new market on the organization record.
What changes on the next market-analysis run
  • /api/market-setup reads the new marketCode.
  • It switches to the new template city, custom event keywords, and seasonal notes.
  • The Marketing and Benchmark agents now research the new market.
  • New MarketEvent and BenchmarkData records are generated for that market context.
What changes on the next event refresh
  • EventIntelligenceAgent.fetchAndCacheEvents() looks up the organization’s current marketCode.
  • It reads the linked MarketTemplate.eventApiConfig and resolves the correct city.
  • The syncer then fetches and caches events for the new city rather than the old one.
What does not happen automatically
  • Old MarketEvent rows are not automatically deleted or archived by the settings update route.
  • Old benchmark data is not automatically regenerated by the settings update route alone.
  • Existing pricing rules are not automatically re-seeded just because marketCode changed in Settings.
  • A plain settings save does not itself rerun the pricing engine.
Recommended client-safe sequence after a market change: save the organization settings, rerun market setup, refresh event feeds, then rerun the pricing engine. This ensures that new market identity, new events, new benchmarks, and new proposals all align with each other.

7. Operational Summary

The simplest way to explain PriceOS to a client

If you need one concise narrative, this is the message.

Step 1: collect truth

PriceOS first pulls portfolio truth from the PMS and market truth from public demand signals and market research agents.

Step 2: build context

The platform assembles listings, inventory, reservations, events, benchmark bands, and guardrails into one explainable operating memory.

Step 3: decide safely

AI agents interpret the context, while the deterministic pricing engine creates day-level proposals inside floor, ceiling, and strategy limits.

Step 4: execute carefully

Only approved proposals are pushed to the PMS, and every write is verified before the system considers it complete.

Step 5: monitor continuously

Status checks, anomaly logic, and guest support agents keep the portfolio operational after pricing changes are live.

Step 6: adapt by market

When the market changes, the organization template changes first, then the research, event sync, benchmark, and pricing layers refresh around the new city.
